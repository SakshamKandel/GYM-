# GYM Tracker — Platform Scale Plan

**Owner:** solo developer / owner
**Status:** master plan (design, not yet implemented)
**Scope:** evolve GYM Tracker from a tier-gated app with an AI coach persona into a role-based platform — a **super admin**, **scoped admins**, and a real **coach dashboard** for Greece — while remaining buildable by one person in phases and leaving the mobile app untouched.

---

## 1. Executive summary

GYM Tracker today has **no role model**. Authentication is an opaque bearer token (`sessions.token → accounts`), resolved by one choke point, `userForToken()` in `apps/web/src/lib/auth.ts`. The only authorization signal is `accounts.tier` (starter/silver/gold/elite), checked inline per route (e.g. `user.tier !== 'elite'` in `api/coach/messages`). "Greece" is an **AI persona** — coach replies are generated by `greeceCoachReply()` (Groq); there is no human coach identity, no admin surface, and the web app renders no UI (its `page.tsx` is a placeholder). Plan videos are **hardcoded into the mobile bundle** (`greeceVideos.ts`), so changing one requires an app-store release.

This plan adds a **role + granular-permission layer additively**: new tables keyed on `accounts.id`, a `permissionsForToken()` helper wrapping the existing `userForToken()`, and a single `requirePermission()` guard that mirrors today's inline tier check. Roles: **super_admin**, four **scoped admins** (member, nutrition, content, support), a **coach** (Greece + future coaches), and the unchanged **end user**. It gives Greece a real dashboard (message assigned users 1:1, view/manage their subscriptions, publish plan videos per tier), moves plan videos out of the bundle into a signed, tier-gated content pipeline (Cloudflare Stream), and stands up the ops spine a multi-staff org needs (audit log, moderation queue, off-provider backups). A mobile user with no staff row behaves **exactly as today** — zero migration risk to end-user auth. Delivered in four phases, each independently shippable, starting free and scaling to ~$5–25/mo.

**One correctness rule threaded throughout:** `accounts.tier` is the auth-read source of truth and `subscriptions.tier` is a parallel copy — every subscription change **must dual-write both** (via one `lib/tier.ts` writer) or the app and billing drift.

---

## 2. Vision & scope — what "scalable like a real company" means here

Big coaching-led companies (FITTR runs 700+ coaches with admins and super-admins; Netflix/Facebook run scoped internal tooling) share four structural traits. Concretely for GYM Tracker:

1. **Identity is separate from entitlement.** A person's *staff role* (what they can administer) is orthogonal to their *consumer tier* (what app features they get). A coach can also be a paying user. We model these on two axes, never conflated.
2. **Least-privilege, scoped roles.** No "one big admin". Work is split into functional domains (members, nutrition, content, support) so a nutrition editor cannot touch billing and a support agent cannot delete accounts. Separation of duties removes single-actor fraud paths ("comp myself Elite *and* stock its videos").
3. **Every privileged action is attributable.** An append-only **audit log** records who did what to whom. This is the accountability spine that makes multi-staff safe.
4. **Coach work is row-scoped, not global.** A coach acts only on their **assigned** users (via `coach_assignments`), so the model scales to hundreds of coaches without any query becoming O(all users).

**In scope:** role/permission layer, super admin + four scoped admins, a coach dashboard, a server-side plan-video pipeline with per-tier gating, and the ops hardening (audit, moderation, backups, caching, rate limits) needed to run staff safely.

**Explicitly out of scope (deferred / non-goals):** real-time chat/websockets (threads stay async, load-on-focus — matches the current app); a full RBAC engine or third-party IAM; rewriting mobile auth; RevenueCat refund-API wiring beyond recording intent (Phase 3+); multi-region infra. The design must stay **buildable by one developer**.

## 3. Role model & permission matrix

Two orthogonal axes:

- **Axis A — consumer entitlement** = `accounts.tier`. UNCHANGED. Governs app features via the existing `hasEntitlement()`.
- **Axis B — staff role** = new, opt-in, absent for ~all accounts. Governs the admin/coach consoles via a new `hasPermission()` / `requirePermission()`.

Roles are explicit permission **sets**, not an inheritance chain. Only `super_admin` can grant/revoke roles — the single privileged escalation path.

```
super_admin              — full control: role grants, billing, danger ops, audit, impersonation
├─ member_admin          — user lifecycle: view/suspend/reset, subscription override, impersonate
├─ nutrition_admin       — diet/nutrition: plans, targets, foods, macro templates
├─ content_admin         — plans/videos/exercises: publish plan videos for ALL tiers
├─ support_admin         — support queue: read/reply support thread; no data writes
└─ coach (Greece + N)    — 1:1 message ASSIGNED users, view their subs, publish videos for own tier scope
end user (implicit)      — no staff row; today's mobile experience, untouched
```

**Permission catalog** — verb-scoped `domain.object.action` strings, stored as data so adding one is a data change, not a schema change:

`members.read` · `members.suspend` · `members.delete` · `subscription.read` · `subscription.override` · `diet.plan.read` · `diet.plan.write` · `diet.food.write` · `content.plan.write` · `content.video.publish` · `content.exercise.write` · `coach.message.user` · `coach.user.read` · `support.thread.read` · `support.thread.reply` · `audit.read` · `impersonate.user` · `roles.grant` · `staff.2fa.manage`

A coach cannot grant themselves users — assignment is a member_admin/super_admin action.

## 4. Greece coach dashboard

A thin web surface under `apps/web` at `/coach`, reusing the existing bearer session (with a browser cookie + 2FA in front — see §7/§9). Greece logs in and sees only **his assigned users**. Every coach route filters by `coach_assignments` server-side; a missing filter would leak all users, so ownership is enforced in the guard, never client-trusted.

**4.1 Assigned users (home).** List of users from `coach_assignments WHERE coach_id = me AND status='active'`, each with tier chip, current streak, last-active, and unread-message count. This is one indexed query and stays flat as coach count grows.

**4.2 Message users 1:1 (inbox + thread).** Today `coach_messages` auto-replies via Groq. The dashboard makes replies **human**:
- Add a nullable `sender_account_id` to `coach_messages` (which coach/admin authored a `coach` row; null = today's AI/system behavior) and a `read_by_coach` flag (the coach's unread badge, mirror of `read_by_user`).
- **Inbox:** assigned users' `coach_chat` threads, unread-first, then by recency; filters for unread / needs-reply >24h.
- **Thread view:** full history for one assigned user; coach replies via `POST /api/coach/threads/[userId]/reply`, which inserts `sender='coach', sender_account_id=greece, read_by_user=false` and marks inbound read.
- **The human reply path must NOT trigger `greeceCoachReply()`** — it is a separate route from the mobile `POST /api/coach/messages`. Additionally, once a user has an active human coach assigned, the mobile POST suppresses the AI auto-reply for that user (a `coach_assignments` lookup). The mobile render path is unchanged: it already shows any non-`user` sender on the left, so human replies appear inline with zero mobile changes.

**4.3 View & manage subscriptions.** For an assigned user: current tier, `expiresAt`, `rcCustomerId`, trial history, referral status. Actions (each audited, permission-limited): grant/extend a trial (respects the existing unique `(accountId,tier)`), and comp/override tier. A plain coach can comp up to **gold**; **elite** comps (limited seats) and overriding paid RevenueCat tiers require `member_admin` or `super_admin`. Every write goes through the single `lib/tier.ts` writer that updates **both** `accounts.tier` and `subscriptions.tier` and records a `subscription_overrides` row (comp grants flagged `rcCustomerId=null` + reason so a future RC reconcile won't clobber them).

**4.4 Plan-video CRUD per tier.** Greece uploads and manages the videos his users watch — his core ask. Video list (thumbnail grid, tier chip, published toggle, remove), plus an upload flow (§8). He sets `tierRequired` per video (the gate) and may attach it to an exercise or a plan. Tier ownership: to keep "own tier" well-defined, Phase 2 restricts coach publishing to a fixed tier scope (Greece owns all tiers as the founding coach); broader per-coach tier ownership is a Phase-4 refinement — until then, `content_admin` handles cross-tier publishing.

**4.5 Coach profile & availability (light).** A `coach_profiles` row (bio, avatar, specialties, `acceptingClients`, `replyWindowHours` default 24, timezone). `replyWindowHours` feeds the auto-ack copy that currently hardcodes "24h". `acceptingClients=false` lets Greece pause new assignments without code changes. The Elite chat header can show his real avatar + "typically replies within Nh".

**Design:** charcoal `#131416` bg, cards one step up (`#1B1D20`, radius 14px, hairline `#26282B` borders), signal-red `#FF3B30` for the primary action and active nav item only, Poppins headings, **Oswald for all numerals** (tier counts, dates, IDs), pill chips for tier/status. No glow, no pulsing dots, no gradient blobs; loading = flat skeleton bars.

## 5. Admin console

A server-guarded `/admin` area inside `apps/web` — the app's first real web UI. A dedicated `/admin/login` (reuses `accounts` + password verify) sets an httpOnly, Secure, SameSite=Lax cookie; a **server-component layout guard** resolves the caller and 404s non-staff before any admin UI renders (the API remains authoritative). Left-rail nav; each item hides when a scoped admin lacks the permission, so UI and API can't disagree.

| Section | Route | Gated by | Backs onto |
|---|---|---|---|
| Overview | `/admin` | any staff | read-only rollups |
| Members | `/admin/members` | `members.read` | accounts, subscriptions, sessions |
| Subscriptions | `/admin/subscriptions` | `subscription.override` | subscriptions, accounts.tier, subscription_overrides |
| Diet & Nutrition | `/admin/nutrition` | `diet.plan.write` | foods, targets, meal plans |
| Content & Plan Videos | `/admin/content` | `content.video.publish` | plans, plan_videos |
| Support Inbox | `/admin/support` | `support.thread.read` | coach_messages (kind=support) |
| Analytics | `/admin/analytics` | any staff (read-only) | aggregates |
| Audit Log | `/admin/audit` | `audit.read` (super only) | audit_log |

**Super admin** sees everything, is the only role that can grant/revoke roles, view the audit log, and impersonate. Bootstrapped once via SQL/seed (no UI mints the first super — chicken-and-egg). **Scoped admins** see only sections their permissions unlock:

- **member_admin** — member list (search by email, filter by tier), detail drawer: change tier (dual-write), **suspend/reactivate** (sets `accounts.status`; `userForToken` gains a `status='active'` filter so suspend instantly kills all sessions), record refund intent (RC API wiring deferred), and impersonate (super/member/support only, read-only, time-boxed, audited).
- **nutrition_admin** — CRUD nutrition plans, macro targets, curated foods.
- **content_admin** — plan-video CRUD across **all** tiers, plan/exercise library edits.
- **support_admin** — support-thread queue (`kind='support'`): read + reply only, zero write access to user data.

Every admin mutation writes an `audit_log` row from day one. `hasPermission()` is the single gate (mirrors `hasEntitlement` per CLAUDE.md rule 4) — no scattered inline role checks.

## 6. Data model (Drizzle, extending the REAL schema)

**Ground-truth constraint that shapes everything:** the schema has a **split identity**. Legacy `profiles.id` (auth-provider subject id) is FK'd by `subscriptions`, `plans` logs, `targets`, `foods`, etc. The auth identity is `accounts.id` (`sessions → accounts`), FK'd by the newer buddy/referral/coach/push tables. **All RBAC and coach tables key on `accounts.id`** — wiring roles to `profiles` would silently break because sessions join to `accounts`. `subscriptions.userId` currently references `profiles.id`; the admin/coach subscription views resolve a user by `accounts.id` and must reconcile to the same person (bridge via email or a one-time backfill — see Risks).

Migrations follow the current workflow: append to `packages/db/src/schema.ts`, then `drizzle-kit push` (Neon HTTP applies `CREATE TABLE IF NOT EXISTS` + `ALTER TABLE ADD COLUMN`). Begin committing generated snapshots (`db:generate`) once RBAC lands, to start a real migration history. All new FKs → `accounts.id`, `onDelete:'cascade'`, `text` PKs with `crypto.randomUUID()` to match existing style.

**New tables**

- **`admins`** — `{ accountId PK → accounts, role enum(super_admin|member_admin|nutrition_admin|content_admin|support_admin|coach), mfaEnrolledAt, totpSecret (enc, nullable), createdAt }`. Presence of a row = staff.
- **`role_permissions`** — `{ role, permission }`, uniqueIndex(role,permission). Seeded from the §3 matrix.
- **`staff_permissions`** — `{ accountId, permission, effect: allow|deny }`. Per-account override overlay; deny wins.
- **`coach_assignments`** — `{ id, coachId → accounts, userId → accounts, status: active|ended, assignedBy → accounts, createdAt }`, uniqueIndex(coachId,userId), index(userId), index(coachId,status). Spine of "assigned users".
- **`coach_profiles`** — `{ accountId PK → accounts, displayName, bio, avatarUrl, specialties jsonb, acceptingClients bool, replyWindowHours int default 24, timezone, isActive, capacity int default 50 }`.
- **`plan_videos`** — `{ id, title, description, exerciseId → exercises (nullable), planId → plans (nullable), tierRequired enum default 'gold', provider enum(cf_stream|bunny), providerVideoId (the CF uid — NEVER a public URL), thumbnailUrl, durationSec, position, status enum(processing|ready|removed) default 'processing', createdBy → accounts, createdAt }`, index(tierRequired,status,position). Replaces the client-hardcoded `greeceVideos.ts` map.
- **`subscription_overrides`** — `{ id, accountId → accounts, tier, reason, grantedBy → accounts, rcCustomerId (null = comp), expiresAt, createdAt }`, index(accountId,createdAt). Append-only log of manual grants; the `accounts.tier`/`subscriptions.tier` columns remain the fast-path source of truth kept in sync by the writer.
- **`audit_log`** — `{ id, actorId → accounts (SET NULL), action, targetType, targetId, meta jsonb, ip, createdAt }`, index(actorId,createdAt), index(targetType,targetId). Append-only; `audit.read` (super) only.
- **`jobs`** — `{ id, type, payload jsonb, runAfter, attempts, status }`. Drained by Vercel Cron; offloads push fan-out, moderation, referral rewards from the request path.

**Column additions (nullable/defaulted — non-breaking for existing rows and the mobile GET/POST)**

- `accounts.status text NOT NULL DEFAULT 'active'` (enum active|suspended) — `userForToken` adds `eq(status,'active')`.
- `coach_messages.senderAccountId text NULL → accounts` — attributes a human coach reply (null = AI/system).
- `coach_messages.readByCoach boolean NOT NULL DEFAULT false` — coach unread badge.
- `coach_messages.flagged boolean NOT NULL DEFAULT false` — moderation flag (async, flag-and-review).
- `sessions.impersonatedBy text NULL → accounts` (Phase 4) — attributes impersonation-minted tokens.

Note: role lives in a dedicated `admins` table rather than an `accounts.role` column, because staff can hold at most one primary role but the override overlay (`staff_permissions`) and audit needs benefit from a real row; a single `accounts.role` column is an acceptable simpler alternative if you never need overlays — but `admins` is the recommended target.

## 7. API surface & authorization middleware

**Extend the one choke point, don't replace it.** In `apps/web/src/lib/auth.ts`, add `permissionsForToken(token)` wrapping the existing `userForToken` — it returns `{ user, role?, permissions: Set<string>, coachUserIds: Set<string> }` (two small indexed queries: role assignment + role→permissions, merged with `staff_permissions` overrides). New `apps/web/src/lib/authz.ts`:

```ts
// Returns a Principal, or a 401/403 Response to early-return — mirrors the
// existing inline `user.tier !== 'elite'` pattern; no middleware, no framework change.
export async function requirePermission(req: Request, perm: string): Promise<Principal | Response>;

// Row-level coach ownership; every /api/coach/[userId] route MUST call this explicitly.
export async function requireCoachOwnsUser(p: Principal, userId: string): Promise<boolean>;
```

Usage in a route (identical ergonomics to today):
```ts
export const runtime = 'nodejs';
export async function POST(req: Request) {
  const p = await requirePermission(req, 'content.video.publish');
  if (p instanceof Response) return p;      // 401/403 already shaped
  // ...p.id is the actor; write plan_videos + logAudit(...)
}
```

`super_admin` bypasses via a **frozen** role-id constant (renaming it would silently drop the bypass on every route). A `logAudit(actor, action, target, meta)` helper is called explicitly inside every state-changing staff route (not global middleware — explicit and testable); a lint/test asserts every `api/admin` mutation calls it.

**New routes** (all `runtime='nodejs'`, `OPTIONS=preflight`, guarded at the top):

*Admin*
- `GET /api/admin/members` — `members.read` — keyset-paginated accounts (search, tier filter)
- `GET /api/admin/members/[id]` — `members.read` — account + subscription + override history
- `PATCH /api/admin/members/[id]` — `members.suspend` — suspend/reactivate
- `PATCH /api/admin/members/[id]/subscription` — `subscription.override` — dual-write tier, insert override, audit
- `GET|POST /api/admin/videos`, `GET|PATCH|DELETE /api/admin/videos/[id]` — `content.video.publish` — plan_videos CRUD
- `GET /api/admin/support`, `POST /api/admin/support/[accountId]/reply` — `support.thread.*`
- `GET|POST /api/admin/roles` — `roles.grant` (super only) — grant/revoke `admins` rows; on revoke, `DELETE FROM sessions WHERE accountId=…` (instant offboard)
- `GET /api/admin/audit` — `audit.read` (super only) — keyset-paginated

*Coach*
- `GET /api/coach/users` / `GET /api/coach/inbox` — `coach.user.read` — assigned users + unread counts
- `GET /api/coach/threads/[userId]` — `coach.message.user` + `requireCoachOwnsUser`
- `POST /api/coach/threads/[userId]/reply` — inserts human coach row, marks inbound read, **bypasses AI**, audit
- `GET|POST /api/coach/subs/[userId]` — view + comp/trial (permission-limited, dual-write, audit)
- `POST|DELETE /api/coach/videos[/id]` — `content.video.publish` (own tier)

*Mobile-facing (video gating)*
- `GET /api/plan-videos/[exerciseId]` — bearer auth; enforce `compareTiers(user.tier, row.tierRequired) >= 0` server-side; mint a signed HLS URL only after the check; `403 {error:'locked', requiredTier}` otherwise.

*Cron*
- `POST /api/cron/drain` (secret-guarded) — process `jobs` batch
- `POST /api/cron/backup` (nightly) — `pg_dump` → Cloudflare R2

**Unchanged:** all existing mobile routes (`/me`, buddy, `/coach/messages` GET, register/login). The existing `POST /api/coach/messages` gains only the "assigned coach → suppress AI" lookup. Reuse `getDb`/`json`/`preflight`/`readJson`/zod exactly. **CORS:** `*` is fine for the mobile bearer API but the admin cookie flow must be same-origin; tighten CORS (origin allow-list) for `api/admin/**` before Phase 3.

## 8. Content delivery (plan videos) & per-tier gating

Videos are hardcoded in `apps/mobile/src/lib/seed/greeceVideos.ts` and played by `expo-video` (native HLS/mp4). Because the URLs ship in the bundle, tier gating is bypassable and every change needs an app release. Fix: move the map into `plan_videos` served by a signed, server-gated API.

**Host:** **Cloudflare Stream** (recommended) — upload mp4 → it transcodes to adaptive HLS, gives per-view signed playback tokens (exactly the tier-gate need), global CDN, ~$0–5/mo at this scale. Bunny Stream is a cheaper equivalent. **Reject** YouTube unlisted (no reliable gating for paid tiers) and raw-mp4-on-S3 (no adaptive bitrate, you'd rebuild transcoding+signing). `expo-video` needs **no change** — it already accepts the signed `.m3u8` URL. Vercel Blob / R2 is for the (separate) progress-photo path only.

**Upload flow (bypasses Vercel Hobby's 4.5 MB body limit — mandatory):** coach picks exercise + file → server requests a Cloudflare **direct-creator-upload** URL → **browser uploads straight to Cloudflare** → CF webhook (or poll) flips `status='ready'` → insert `plan_videos` storing only `providerVideoId`. Remove = soft-set `status='removed'` + CF delete (never hard-delete; audit).

**Per-tier gating — enforced at the URL-mint boundary, not by hiding cards.** `GET /api/plan-videos/[exerciseId]` looks up the ready row, checks `compareTiers(user.tier, tierRequired) >= 0` (reuse `entitlements.ts`, don't reimplement), and only then mints a Cloudflare signed token (~2 h TTL). Store the provider id, never the playback URL, so links can't be shared. Elite "form check" videos map to the existing `form_checks` entitlement.

**Mobile change (minimal):** replace the local `getGreeceVideo(id)` lookup with a `useQuery` to `/api/plan-videos/[id]`; on 403 render the existing locked/paywall card, on 200 pass `url` to the unchanged `ExerciseVideo`. Cache the response ~1 h client-side — **the signed-URL TTL (2 h) must exceed the client cache TTL** — and refetch on a 401/expired playback error. Keep `greeceVideos.ts` as a fallback seed for one release, then delete it.

## 9. Scalability, ops, security & moderation

- **Multi-coach scale (FITTR shape).** Every coach query is row-scoped through `coach_assignments` (`coachId + status` indexed), so it stays O(assigned users), flat as coaches grow. `coach_profiles.capacity` lets auto-assignment pick the least-loaded active coach. When inbox listing gets heavy, add a `coach_thread_summary` read-model (last message + unread count, updated on write) instead of N thread scans.
- **Permission caching.** `permissionsForToken` runs two small queries per guarded request. Cache the role→permission set in a short-TTL (≤60 s) in-instance LRU — Neon HTTP instances are short-lived, so no session-wide cache; a revoked role clears within the TTL, and session-revoke-on-offboard is instant regardless.
- **Background jobs.** Push fan-out and AI replies run inline today. Introduce the `jobs` table drained by **Vercel Cron** (`/api/cron/drain`, every 1–5 min): enqueue push/moderation/referral work so slow calls don't block requests or couple them to FCM/Groq uptime.
- **Caching.** Static reads (exercises, plan metadata) get `Cache-Control: s-maxage` (Vercel edge, free) — also protects the Neon compute budget since Neon HTTP has no pool.
- **Rate limiting.** None today. Add a fixed-window limiter keyed by `accountId` on write routes (coach POST, uploads, AI tip) — in-DB counter or Upstash free tier — before high-value staff accounts exist.
- **Audit & impersonation.** Every state-changing staff action writes `audit_log`. Impersonation (super/member/support only) is **read-only** (write routes reject when the `impersonatedBy` marker is present — a fail-closed check in the guard, not per-route opt-in), time-boxed (~30 min), and both start and stop are audited. Defer to Phase 4; ship without it first.
- **2FA for staff.** Mandatory TOTP for all staff roles (never end users). A staff account with a role but no `mfaEnrolledAt` cannot pass `requirePermission`. super_admin can reset (audited, ideally two-person). Add login attempt throttling alongside.
- **Moderation & PII.** Run an async moderation pass (jobs queue) on every user-authored `coach_messages.body` — keyword denylist first, optional LLM for flagged items — set `flagged`, surface in the admin queue; never block the write. Keep progress photos and videos in **private** storage served only via short-TTL signed URLs. Don't log message bodies or emails (log `accountId` only). Provide a GDPR-shaped hard-delete path (cascades already exist via FKs).
- **Backups (highest-priority gap).** Neon PITR retention on Hobby is short and single-provider. Add a nightly off-provider `pg_dump → Cloudflare R2` (`/api/cron/backup`) so a Neon incident or an accidental `drizzle-kit push` isn't data loss. Do this in Phase 0.
- **Observability.** Replace scattered `console.error` with a structured logger (pino) + Sentry free tier; log `{route, accountId, ms, status}` per call.

## 10. Phased roadmap

Each phase is independently shippable and leaves mobile untouched. Effort assumes one developer.

### Phase 0 — De-risk & foundation (weekend, ~$0)
- Nightly off-provider Neon backup (Vercel Cron → R2). **Highest-priority gap.**
- Sentry free tier + structured logging.
- Shared tier-rank usage tightened (reuse `compareTiers`); replace the literal `user.tier !== 'elite'` in `api/coach/messages` with the helper.
- No user-facing change.

### Phase 1 — Role plumbing + super admin (≈0.5–1 wk, ~$0)
- Tables: `admins`, `role_permissions`, `staff_permissions`, `audit_log`; seed the §3 matrix; seed the owner as `super_admin` via SQL (guarded seed, refuses to double-run).
- `permissionsForToken` (extend auth.ts) + `requirePermission` (new authz.ts) + `logAudit`.
- `accounts.status` column + `userForToken` active filter.
- `/admin/login` (cookie), server-component layout guard, and a minimal `/admin` overview + one proof route (`/api/admin/whoami`).
- **Unblocks everything; ships in a weekend.**

### Phase 2 — Coach dashboard (≈1–1.5 wk, ~$0)
- `coach_assignments`, `coach_profiles`; add `senderAccountId` + `readByCoach` to `coach_messages`; seed Greece as `coach`.

### Phase 3 — Scoped admins + video pipeline + subscription mgmt (≈2–3 wk, ~$0–5/mo)
- Scoped admins: member_admin (suspend, subscription.override), content_admin, nutrition_admin, support_admin; full matrix live; tighten CORS on `api/admin/**`.
- `subscription_overrides` + single `lib/tier.ts` dual-writer; audited comp/override/trial flows.
- `plan_videos` + Cloudflare Stream + direct-creator-upload + signed `GET /api/plan-videos/[id]`; coach/admin upload UI; mobile swaps the local map for the API. **The differentiated content library + per-tier video gating.**
- Read-only impersonation (basic).

### Phase 4 — Scale & hardening (≈1–2 wk, ~$0–10/mo)
- `jobs` table + Vercel Cron drain (move push fan-out, moderation, referral rewards off the request path).

## 11. Risks & open decisions

1. **`accounts.tier` vs `subscriptions.tier` (sharpest trap).** Auth reads `accounts.tier`; the app also reads `subscriptions`. Every override MUST dual-write both, wrapped in one `lib/tier.ts` writer, or access diverges from billing.
2. **`profiles` vs `accounts` split.** RBAC/coach tables key on `accounts.id`, but `subscriptions.userId`/`plans` logs FK `profiles.id`. Admin subscription views must reconcile a person across both ids (bridge by email or a one-time backfill). **Open decision:** unify on `accounts.id` long-term vs. maintain a bridge. Never hang roles off `profiles`.
3. **Migration history.** Repo ships via `drizzle-kit push` with no committed migrations; verify the first `db:generate` snapshot applies cleanly (existing tables show "already exists"). Switch to reviewed migration files before multiple envs/coaches touch prod.
4. **Human reply must not trigger the AI.** The coach reply route must be a separate path from the mobile POST, or every coach message gets a bot echo. Also gate the mobile POST on an active assignment.
5. **Coach row-scoping is a second, non-automatic check.** Any `/api/coach/[userId]` handler that forgets `requireCoachOwnsUser` leaks all users' threads. Enforce by convention + a test.
6. **super_admin bypass id must be frozen.** Renaming the `super_admin` role id silently removes the bypass on every route. Keep it a constant.
7. **Video gating at the mint boundary + TTL ordering.** Enforce tier at signed-URL mint, not by hiding cards. Signed-token TTL (2 h) must exceed client cache TTL (1 h) or playback 401s mid-session; refetch on expiry.
8. **Vercel 4.5 MB body limit** makes server-proxied uploads impossible — direct-creator-upload is mandatory.
9. **Storage not yet wired.** `photos.storagePath` is a stub; standing up a private bucket + signed URLs is a Phase-3 prerequisite for videos, not a given.
10. **Impersonation is the highest-risk feature** — read-only, fail-closed, time-boxed, always audited; defer to Phase 4 rather than ship half-baked.
11. **CORS** `*` is fine for mobile bearer but must be tightened (same-origin cookie + allow-list) for `api/admin/**` before scoped admins exist.
12. **Bootstrap** the first super_admin has no UI (chicken-and-egg) — document the one-off SQL/seed or risk lockout.

## 12. Reality check — build this first (solo-dev cut)

The full four-phase design above is the *destination*. As a one-person company you should treat Phases 3–4 as an **on-demand backlog, not a front-to-back roadmap** — most of it defends multi-staff / high-scale problems you don't have yet. Ship the two things Greece actually asked for, with a few non-negotiable safety rails, and split the rest the day a second human or real load appears.

### MVP — ~2–3 focused weeks, delivers 100% of Greece's ask
1. **Human coach messaging.** Add nullable `coach_messages.senderAccountId` + `readByCoach`; a `coach_assignments` table; one cookie-guarded `/coach` route group; inbox of assigned users → thread view → `POST /api/coach/threads/[userId]/reply` that inserts `sender='coach'` and **does not** call `greeceCoachReply()`. Add the "active human coach assigned → suppress AI auto-reply" lookup to the existing mobile POST. **Zero mobile changes** (the chat already renders any non-`user` sender on the left — *verify in the actual mobile component first*).
2. **Per-tier plan videos.** `plan_videos` table; commit to **Cloudflare Stream**; direct-creator-upload (mandatory — dodges Vercel's 4.5 MB body limit); `GET /api/plan-videos/[exerciseId]` that runs `compareTiers(user.tier, tierRequired) >= 0` (reuse `entitlements.ts`) **before** minting a ~2 h signed URL. Mobile swaps `getGreeceVideo(id)` for a fetch; renders the existing locked card on 403.
3. **Minimal auth spine to make it legal:** an `admins` table (`accountId PK, role`) whose presence = staff; seed Greece as `coach` via SQL; a single `requirePermission()` / `requireCoachOwnsUser()` pair in a new `authz.ts` wrapping `userForToken()`. **Hardcode** the coach's allowed actions for v1 — no `role_permissions`/`staff_permissions` engine yet.

### Non-negotiable safety rails — even in the MVP
- **Audit log from the first privileged route** (not Phase 4). `{actorId (ON DELETE SET NULL), action, targetType, targetId, meta, ip, createdAt}`, append-only, `audit.read` super-only. Cheap now, painful to backfill.
- **Coach row-scoping on every `/api/coach/[userId]` route** via `requireCoachOwnsUser()`, enforced by convention **plus a test** that hits a non-assigned userId and asserts 403. A forgotten filter leaks every user's private threads.
- **Tier enforced at the signed-URL mint boundary**, never by hiding cards. Store `providerVideoId`, never a public URL. Signed-token TTL (2 h) **>** client cache TTL (1 h).
- **Suspend must instantly kill sessions:** add `accounts.status` + `eq(status,'active')` inside `userForToken()`; on role revoke `DELETE FROM sessions WHERE accountId=…`.
- **2FA (TOTP) before the console can suspend accounts or move tiers.** If 2FA can't ship with the MVP, the MVP console is **read + message only** (no suspend, no paid-tier override) until it does.
- **Nightly `pg_dump → R2` backup before pointing a human at prod data.** One cron route, an afternoon; removes the scariest single-provider data-loss risk.

### Deliberately deferred (build when it earns its keep)
Four scoped admin roles → ship **super_admin + coach only**, split the rest at hire #2. The data-driven permission engine (`role_permissions` + `staff_permissions` "deny wins" + 20 verb strings) → a hardcoded `switch(admins.role)` for two roles, refactors into the matrix in an afternoon later. `jobs` queue + cron drain, `coach_thread_summary` read-model, LRU permission cache, Upstash rate limiting, `coach_profiles.capacity` auto-assignment (you have **one** coach — assign by hand), async LLM message moderation → all speculative at low-hundreds of users; defer until Sentry shows real latency or a second coach exists.

### Decisions to lock before writing code
1. **Video host** — Cloudflare Stream (recommended; signed per-view tokens are exactly the tier-gate primitive) vs Bunny.
2. **`profiles.id` vs `accounts.id` reconciliation** — `subscriptions.userId` FKs `profiles.id`, auth/coach tables key on `accounts.id`. One-time backfill to unify, or an email bridge? Every "view this user's subscription" screen depends on it — sharpest correctness trap.
3. **Admin/coach auth** — reuse `accounts` password + httpOnly/SameSite cookie + server-component guard (recommended) vs a separate credential store. Decides the CORS story (`*` bearer for mobile must **not** apply to cookie'd admin routes).
4. **`lib/tier.ts` dual-writer contract** — `accounts.tier` (auth source of truth) and `subscriptions.tier` (parallel copy) must be written together, or app access and billing drift. Lock the signature before any comp/trial UI.
5. **Paid-tier overrides** — can the coach override a real RevenueCat subscription in v1, or only issue free comps (`rcCustomerId=null`)? Ship the narrow version first.
6. **When 2FA becomes mandatory** — decide the trigger consciously; don't let it default to "never got to it."

### Consistency notes vs the codebase (fix during Phase 0/1)
- `CLAUDE.md` rule 3 ("every new table ships with RLS") — the real code uses **API-layer** authorization, not RLS. Amend the rule to "API-layer authorization via `requirePermission()`" so the RBAC layer is the documented enforcement boundary.
- `CLAUDE.md` rule 4 (tier gating **only** via `hasEntitlement()`) — `api/coach/messages` uses a raw `user.tier !== 'elite'` literal, a live violation. Route it through the helper during Phase 0 and reuse `entitlements.ts` for the new video gate.

**Verdict:** *Architecturally sound and correctly grounded in the real code — build the two MVP slices (human coach messaging + Cloudflare-signed tier-gated videos) with audit log, session-killing suspend, and coach row-scoping from day one, and defer the four-role permission engine, jobs queue, impersonation, and scale plumbing until a second human or real load actually exists.*
