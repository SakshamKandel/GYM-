*Head of Product + Principal Engineer synthesis of 18 journey-walk reports. Tree = v1.0.6, all gates green. Verified against live source: schema (`packages/db/src/schema.ts`, 2267 lines, all tables enumerated), push layer (`apps/web/src/lib/push.ts` — `sendPushToAccount`, `devicePushTokens` table), `accounts.tier*` columns (already present: `tierStartedAt/tierExpiresAt/tierSource/tierSourceId`), coach payout rail (`walletLedger`, `coachPayoutRequests`), and every route/screen path cited below.*

## 1. Executive Summary — the 5 themes

The 18 journeys are, individually, well-engineered happy paths. Money is server-authoritative, state machines are CAS-guarded and race-safe, PII is masked, deletions are fail-closed. **Almost nothing on the "core loop" is broken.** The failures cluster into five systemic gaps that a professional product closes and a prototype leaves open:

**T1 — The post-purchase relationship is absent.** Every commerce flow (meal order, meal subscription, tier purchase) nails the *transaction* and abandons the *customer* the instant money moves. No confirmation screen, order number, ETA, receipt/invoice, rating, or tip. "A hungry customer is left refreshing a list to learn what's happening." This repeats verbatim across Orders, Subscriptions, Tiers, and even Gyms (dead-ends after "Call/Directions").

**T2 — Notifications are one-directional and everything else is pull-based.** Push fires *only* when staff/coach act *on* a member. The partner is never told an order arrived (fatal for a delivery product). Staff are never told a receipt/support-ticket/application landed. The coach is never told a client messaged. Even the member is missing the entire *scheduled/async* class: trial expiry, subscription-cycle dunning, renewal nudge, day-2 re-engagement. There is no notification center, no preferences, no polling/realtime on live surfaces (order tracking, menu, staff queues).

**T3 — Money is computed correctly but never reconciled or made whole.** The partner "digital held" figure is a live SQL sum that *never decrements* — after a real off-platform payout it stays inflated forever and reads as permanently-owed. There is **no partner payout rail at all** (coaches have the complete rail: `walletLedger` + `coachPayoutRequests` + admin queue + wallet UI). On the tier side, expiry/renewal/cancel/proration semantics don't exist. Members can't see invoices, receipts, or payment history anywhere.

**T4 — Shipped engines have no front door; feature reach is split across platforms.** The subscription **EDIT engine is fully built server-side and unreachable in the app**. Coach client-management (assign workout/diet, milestones, tier grants) exists **only on mobile** — a coach at a desktop is locked out of the two core paid deliverables. Gym ratings render as social proof with **no write path**. The staff hub fetches `overview.ops` badge counts and **never renders them**. Pattern: backend capability exists, UI/entry-point missing.

**T5 — Silent failures and dead-end error states.** Unguarded async saves hang forever (food search, copy-yesterday) with no error surface. Cancel buttons show for orders the server *always* 409s (guaranteed dead-end tap). "Contact support" is plain text with no link. Price changes are charged silently. A deleted meal maps to the wrong error copy. Coach-unassign is silent. Across the board, error paths lack both an explanation of *why* and a route *forward*.

**The v1.1 mandate:** close all five. Ship the notification backbone + scheduler, the partner payout rail, the full post-purchase relationship (confirm → receipt → rate → tip → dispute), the tier lifecycle, cross-platform coach parity, and harden every silent-failure edge.

**Global ownership rules (enforced across all packages):**

**Frozen contract (all push senders use this):**
```
notify(event: NotificationEvent, target:
   { accountId: string } | { role: 'staff', permission: Permission } | { partnerId: string },
   payload: { title: string; body: string; data?: { type: string; id?: string } })
   → Promise<void>   // best-effort, never throws; writes inbox row; gated by prefs+quiet-hours
```
- `data.type` is the mobile deep-link key (`'order'|'cycle'|'tier'|'coach_chat'|'support'|'gym'|…`), `data.id` the target id. Mobile deep-link switch is owned by WP-14 (`lib/notifications.ts`).
- **`notify` is fire-and-forget from every call-site** (`void notify(...)`) — the inbox DB insert and the FCM send both live inside `notify`'s own try/catch; it MUST NOT be `await`ed on a route's critical path, so a Neon pool-exhaustion or FCM outage can never fail the business mutation that triggered it (§7.1). The inbox row is written first (durable) with `sentAt=null`; push send flips `sentAt`; `retry-unsent` cron reconciles.
- Cron routes are idempotent: they set `notifications.dedupeKey` (e.g. `trial_expiry:{accountId}:{yyyy-mm-dd}`) so the partial-unique insert makes a re-run / Vercel double-fire a no-op — never a double-notify. Mass-scan crons (`day2-reengage`, `renewal-nudge`) **page the account scan** (batches of ≤500, capped per run) rather than loading the whole table. **No new infra** — Vercel Cron hitting these handlers. SMS/email channels are **§6-deferred**; `notify` is push+inbox today.
- **Injection defense (§7.2-S2):** `title`/`body` are **server-templated**. When a notification echoes user-authored text (delivery/order notes, dispute notes, coach or gym-enquiry messages) into a **privileged** recipient's (staff / partner / coach) notification, it MUST be `maskPii`'d and attributed ("Member note: …"), never rendered as platform-authored — a member must not be able to forge a "Your payout is ready"-style staff/partner notification through a free-text field. Quiet-hours suppresses the **push** only; the inbox row is always written (no end-of-window push storm — the inbox is the durable record).
